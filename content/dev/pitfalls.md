Read this twice. Each entry is a real failure mode that has either
hit this project or hits Ren'Py + AIO projects in general.

## Audio and accessibility

**Don't:** `play music garden_grove fadein 2.0`
**Do:** `$ play_music("garden_grove", fadein=2.0)`

The helper at `accessibility.rpy:~140` calls `renpy.notify()` with the
caption registered for that track. Bypassing it means caption-users
get silence. Same rule for `play_sound()`.

**Don't:** `with hpunch`
**Do:** `$ shake("h")`

`shake()` respects `persistent.no_shake`. `with hpunch` does not.

**Don't:** define an image without a caption.
**Do:** in `accessibility.rpy`, add an entry to `IMAGE_CAPTIONS`:

```python
define IMAGE_CAPTIONS["bg dark_alley"] = _("a narrow alley between leaning buildings, lit by a single guttering lantern")
```

Then `scene bg dark_alley` will fire the caption automatically.

## Ren'Py syntax traps

**`default` vs `define`.** `default` is for *variables that change at
runtime* (game state). `define` is for *constants* (characters,
images, screens). If you `define x = 0` and try to `$ x += 1`, it
silently no-ops at runtime — `define` values are baked at compile.
Use `default` for any stat counter.

**`$` vs `python:` blocks.** Single-line Python expressions use `$`.
Multi-line use `python:`. Don't try to do an `if` block on one line
with `$ if x: y = 2`; it will parse but behave unpredictably.

**No `async`/`await`.** Ren'Py 8 is Python 3 but the script is
synchronous. Don't use coroutines. Use Ren'Py screens and labels
for anything that "waits."

**Persistent vs default.** `default x = 0` resets per save.
`persistent.x = 0` survives across all saves. The achievements
framework uses persistent. Don't accidentally store
`ryoka_corruption` as persistent — it must be per-save.

**`init python` runs at compile.** Anything in `init python:` runs
once when the game launches, before any save loads. Don't put
game-state modifications there.

## AIO template specific

**`definitions.rpy` is dormant.** It tries to rewrite itself in
binary mode. Ren'Py 8 refuses. The flag `redefine_resources = False`
keeps it harmless. **Do not edit this file.** If you need to
auto-discover assets, use Ren'Py 8's built-in image directory
scan, not the template's old custom one.

**`splashscreen` flow is multi-stage.** It chains:
splash → first-run accessibility prompts → content_warning → main
menu. If you `jump start` directly from `splashscreen`, you skip
the content warning. Always end `splashscreen` with `return` and
let the template's flow handle the rest.

**`skip_splashscreen` exists.** If a player has set the "skip splash"
preference, the template uses that label. Don't break it.

**Eileen LayeredImage is referenced internally.** The template's
demo gallery / replay screens may reference Eileen images. Even if
you remove all Eileen content from `script.rpy`, **leave the
LayeredImage definition** in place until you're ready to fully
audit `extras.rpy` and `screens.rpy`.

**Achievement definitions are list-based, not dict-based.** Look at
`BOBCACHIEVEMENT_LIST` in `0bobcachievements.rpy`. Each entry is
a list `[name, internal_id, description, icon, hidden]`. To add
yours, replace this list — don't try to mutate it at runtime.

**The first-run accessibility prompt is gated by
`persistent.first_run`.** Don't unset that flag in code or every
launch will re-prompt the user.

## Save-system traps

**`config.save_directory` collides with the template's default
unless you change it.** We did, but if you ever copy this template
to a new project, *change it again* — sharing a save dir between
two games corrupts both.

**Saves embed the script tree.** If you rename a label after a
player has saved at that label, their save breaks. Plan label
names early. Once published, label names are forever.

**Persistent vars survive renames inconsistently.** If you rename
`persistent.ryoka_corruption` to `persistent.ryoka_corruption_v2`,
old saves still have the old key. Either migrate or accept loss.

## Combat / menu system traps

**Ren'Py `menu:` blocks have hard limits.** You cannot easily build
a tree (submenu) with the built-in `menu:`. For Sabotage's
4-option submenu, use a custom `screen sabotage_picker():` and
`call screen` — *not* nested `menu:` blocks.

**`menu:` choice text is statically compiled.** You cannot
interpolate `f"{x} ({y} gold)"` into a choice label without using
a `python:` block to build the menu items dynamically. Ren'Py
docs cover the dynamic menu pattern.

**Disabling a menu choice = greying it out, not hiding it.**
Disabled choices still show. Use `if` guards if you want to *hide*
an option entirely. For sabotage gating, *show but disable* is
the design — the player must see what they're missing.

## Dialogue / variable interpolation

**`[mc_name]` interpolates inside strings; `{mc_name}` does not.**
Ren'Py's interpolation is `[brackets]`. Curly braces are for text
tags (color, size, etc.).

**`"Don't"` in dialogue.** Use the apostrophe `'`, not the curly
quote `’`, or Ren'Py may complain about encoding under some
locales. Same for ellipsis: `...` not `…`.

**Triple-quoted strings preserve indentation.** If you write
multi-line dialogue, dedent properly. Leading spaces become part
of the displayed text.

## Performance / scope traps

**Don't load all images at once.** Use Ren'Py's automatic image
caching. If you `image ryoka_50_expressions = ...` for 50 expressions
and only use 5, they all load on game start.

**The template's gallery / music room screens iterate large lists.**
If you add 200 fragments to the music room, the screen rebuild on
every interaction will lag. Paginate.

**Saves get large fast.** Every `default list = []` is serialized
per save. If you `default combat_log = []` and append every action,
saves balloon. Cap with `combat_log = combat_log[-50:]` after writes.

## Project-specific traps

**Splitting `party_corruption` was a deliberate design choice.**
If you see a request "use shared corruption" — refuse. The wiki's
Crucible logic depends on per-girl values.

**Six endings, not three.** Don't simplify. Don't merge "Vanilla
Harem" and "Vanilla One" — they're different acceptance criteria.

**Tragedy is the default.** If `route_locked is None` at Day 90,
the player gets Tragedy. Don't add fallbacks that "rescue" the
uncommitted player. The deadline is the design.

**MIT credits are non-negotiable.** Don't simplify the about text
to drop the AIO contributors. The template is MIT-licensed, which
permits commercial use, but requires attribution.

**The protagonist's level being 50 vs the girls' 99/95 is not a
bug.** It's a comedy beat. Don't "fix" the math.

## When in doubt

1. Read the AIO file you're touching, in full.
2. Check [Variables](/wiki/dev/variables) and [Labels](/wiki/dev/labels).
3. Boot the game and reproduce the issue with one debug printout.
4. If still stuck, surface the question in the format
   "I see X. The wiki says Y. The code does Z. Recommend A because B."

Don't guess silently.
